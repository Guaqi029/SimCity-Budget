# 《模拟城市记账本》团队分工与技术选型说明

## 1. 团队开发模式

本项目由 2 人团队协作完成，采用 **“模块分工为主，前后端分工为辅”** 的开发模式。

由于《模拟城市记账本》是一个无云端、本地化运行的 HarmonyOS 应用，系统中的页面展示、用户交互、业务逻辑、本地数据存储和健康度算法之间联系较为紧密。如果完全按照传统“前端 + 后端”方式分工，容易出现页面等待接口、逻辑等待页面、联调成本较高等问题。

因此，本项目更适合按照功能模块划分职责。每名成员负责一个相对完整的功能域，包括该模块下的页面 UI、数据模型、业务逻辑、Service 调用、本地存储对接和测试工作。

本项目的分工方式如下：

| 成员   | 负责模块    | 主要职责                               |
| ---- | ------- | ---------------------------------- |
| 成员 A | 账单与记账模块 | 负责新增账单、账单列表、账单详情、账单编辑删除、账单数据存储与查询  |
| 成员 B | 账号、预算与小镇模块 | 负责账号管理、预算设置、分类看板、首页小镇、健康度算法、统计计算与视觉状态切换 |

可以在项目文档中表述为：

> 本项目采用模块化协作开发模式。团队不完全按照传统前后端方式划分任务，而是以功能模块为单位进行职责分配。每名成员负责对应模块的页面展示、交互逻辑、数据处理和测试工作，同时通过统一的数据模型、Service 层和 Repository 层保持项目结构规范。由于系统采用本地端侧架构，不部署独立服务器，所有数据存储、业务计算和页面展示均在 HarmonyOS 应用内部完成。

### 2.2 开发原则

本项目遵循以下开发原则：

1. **模块职责清晰**：每个成员负责相对独立的业务模块，减少相互等待。
2. **统一数据模型**：所有模块共用 `BillRecord`、`BudgetConfig`、`Category`、`CityState` 等模型。
3. **页面不直接操作数据库**：页面通过 Service 层调用业务方法。
4. **业务逻辑集中封装**：预算计算、健康度判断、账单统计等逻辑集中放在 Service 或 Calculator 中。
5. **本地优先**：应用不依赖网络服务，所有数据均存储在设备端。
6. **可测试可维护**：核心算法、数据统计、增删改查逻辑应便于单独测试。

## 10. 工作边界说明

### 10.1 页面与数据库边界

页面不直接操作数据库。

不推荐：

```typescript
// 不推荐
BillComposer 直接操作 RelationalStore
```

推荐：

```typescript
// 推荐
BillComposer 调用 BillService
BillService 调用 BillRepository
BillRepository 操作 RelationalStore
```

---

### 10.2 模块之间通过 Service 通信

账单模块和预算模块之间不直接访问对方页面，而是通过 Service 方法交换数据。

例如：

```typescript
// 预算模块需要账单数据
StatisticService 调用 BillService.getBillsByMonth(month)
```

```typescript
// 账单模块需要判断分类预算预警
BillComposer 调用 BudgetService.checkSingleBillWarning(month, categoryId, amount)
```

---

### 10.3 视觉表现与算法结果分离

健康度算法只返回等级和说明，不直接控制 UI。

算法返回：

```json
{
  "level": 3,
  "name": "萧条",
  "description": "累计消费已超过当前合理进度线"
}
```

首页根据 `level = 3` 展示：

```text
破旧木屋
阴天天气
灰色滤镜
落叶动画
```

## 14. 可以写进报告的简洁版

> 本项目由 2 人团队完成，采用“模块分工为主，前后端分工为辅”的协作模式。由于本系统是无云端的 HarmonyOS 本地应用，页面展示、业务逻辑、本地存储和状态刷新之间联系紧密，因此团队不采用完全传统的前后端分工，而是按照功能模块划分职责。
>
> 成员 A 负责账单与记账模块，包括首页记账区、账单列表、账单详情、账单编辑删除、账单数据模型设计、RelationalStore 数据库存储和账单查询统计等内容。成员 B 负责账号、预算与小镇模块，包括首页小镇展示、账号管理原型、预算设置、统计分析页（含分类预算看板与月度统计）、Preferences 预算存储、四层动态健康度算法和小镇视觉状态切换等内容。
>
> 两名成员通过统一的数据模型、Service 层接口和 Repository 层封装进行协作。页面不直接操作数据库，业务逻辑集中封装在 Service 层，数据读写由 Repository 层负责。该分工方式既保证了模块职责清晰，又减少了等待接口和联调阻塞的问题，适合 2 人团队在 3 周开发周期内完成项目。

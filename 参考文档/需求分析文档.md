### 1.2 项目背景

传统记账软件通常以表格、数字、账单列表和统计图为核心，虽然功能完整，但对用户的长期使用吸引力不足。许多用户在开始记账后，容易因为反馈不明显、过程枯燥、短期收益低而放弃使用。

《模拟城市记账本》尝试将个人财务管理与轻量化模拟经营玩法结合。用户的消费行为会实时影响虚拟小镇的建设状态：理性消费会让小镇繁荣发展，超支消费会导致小镇萧条甚至破产。系统通过“视觉反馈 + 财务算法 + 预算管理”的方式，引导用户形成更健康的消费习惯。

---

### 1.3 产品定位

《模拟城市记账本》是一款面向个人用户的本地化轻量记账应用，主要运行于鸿蒙系统设备端。

产品定位如下：

| 项目   | 描述                                  |
| ---- | ----------------------------------- |
| 产品类型 | 个人财务管理 + 轻量模拟经营                     |
| 目标平台 | HarmonyOS                           |
| 目标用户 | 有记账需求、预算控制需求或希望通过趣味方式培养消费习惯的学生与年轻用户 |
| 核心价值 | 将消费控制结果可视化、游戏化，提高用户记账积极性            |
| 数据策略 | 完全本地存储，不依赖云端服务                      |
| 开发周期 | 约 3 周                               |
| 团队规模 | 2 人开发团队                             |

---

### 1.4 产品目标

本系统的核心目标包括：

1. 支持用户设置每月总预算与分类预算。
2. 支持用户记录收入、支出、消费类别和日期。
3. 支持展示当月账单流水和分类预算消耗情况。
4. 基于“日均预算线”和“累计消费线”判断用户财务健康度。
5. 将财务健康度实时映射为小镇视觉状态。
6. 通过正向激励和负向预警提高用户对消费情况的感知。
7. 在不使用网络服务的前提下，完成完整的记账、预算、统计和可视化反馈闭环。

---

### 1.5 产品理念

本产品的核心理念是：

> 将枯燥的财务管理转化为正向反馈的城市建造体验。

系统通过追踪用户的单日消费、累计消费、月预算消耗情况和分类预算消耗情况，将抽象的财务状态转化为直观的小镇状态。

用户不再只是看到“今天花了多少钱”，而是看到自己的消费行为如何影响一座小镇的繁荣、平稳、萧条或破产。通过这种游戏化反馈机制，系统希望帮助用户建立“细水长流、理性消费”的习惯。

## 4. 用户角色与使用场景

### 4.1 用户角色

本系统主要面向单一普通用户角色。

| 用户角色 | 描述                      | 权限                              |
| ---- | ----------------------- | ------------------------------- |
| 普通用户 | 使用应用进行预算设置、记账、查看账单和小镇状态 | 可新增、查看、编辑、删除账单；可设置预算；可查看统计与城市状态 |

---

### 4.2 典型用户场景

#### 场景 1：月初设置预算

用户在每月初打开应用，输入本月总预算，例如 2000 元。用户还可以根据自己的消费习惯设置分类预算，例如餐饮 800 元、娱乐 300 元、学习 300 元、日用 400 元。

系统保存预算后，根据当月天数自动计算日均预算。

---

#### 场景 2：每日记账

用户晚饭后打开应用，记录一笔餐饮支出 35 元。系统保存账单后，立即重新计算当天支出、本月累计支出和财务健康度，并刷新小镇状态。

如果用户消费控制良好，小镇呈现繁荣或平稳状态；如果累计消费超出当前合理均线，小镇进入萧条状态。

---

#### 场景 3：分类预算预警

用户设置娱乐预算为 500 元，但某次记录娱乐消费 600 元。系统检测到单笔金额超过娱乐分类预算，立即触发局部雷暴或警告动画，提醒用户该消费行为风险较高。

---

#### 场景 4：预算超支导致破产

用户本月总预算为 2000 元，但累计支出达到 2050 元。无论当天消费是否较低，系统都会将小镇状态判定为第四层“破产状态”，展示废墟、雷暴、雨滴和屏幕震动效果。

---

#### 场景 5：查看分类消费情况

用户进入分类看板，查看餐饮、娱乐、学习、日用等类别的预算消耗进度。系统通过进度条展示每个类别的已消费金额、预算金额和消耗比例。

---

## 5. 总体业务流程

### 5.1 主流程

1. 用户首次进入应用。
2. 系统检测是否已设置当月预算。
3. 如果未设置预算，进入预算设置页面。
4. 用户输入月总预算，可选输入分类预算。
5. 系统保存预算配置。
6. 用户进入首页。
7. 首页展示当前小镇状态、今日消费、本月累计消费、剩余预算和健康度说明。
8. 用户点击记账按钮。
9. 用户填写金额、类型、类别和日期。
10. 用户提交账单。
11. 系统校验输入合法性。
12. 系统保存账单到本地数据库。
13. 系统重新计算健康度。
14. 系统刷新小镇 UI。
15. 系统更新账单列表和分类预算进度。

---

### 5.2 记账流程

```text
进入记账页面
   ↓
填写金额、收支类型、类别、日期
   ↓
点击保存
   ↓
校验金额是否有效
   ↓
校验类别是否存在
   ↓
写入本地数据库
   ↓
计算 E_day、E_total、D、T
   ↓
判断健康度层级
   ↓
判断是否触发分类预算预警
   ↓
刷新首页小镇状态与账单数据
```

---

### 5.3 预算设置流程

```text
进入预算设置页
   ↓
输入月总预算
   ↓
选择是否设置分类预算
   ↓
填写各类别预算
   ↓
校验预算金额合法性
   ↓
保存到 Preferences
   ↓
重新计算日均预算
   ↓
刷新财务健康度
```

# 6.1 预算管理模块

## FR-001 月总预算设置

### 需求描述

系统应支持用户设置当前月份的月总预算。月总预算是财务健康度算法的核心输入参数。

### 功能规则

1. 月总预算为必填项。
2. 月总预算必须大于 0。
3. 月总预算最多支持两位小数。
4. 用户可在任意时间修改月总预算。
5. 修改月总预算后，系统应立即重新计算当前月份的财务健康度。
6. 每个月应对应独立的预算配置。
7. 当进入新月份时，如果未设置新月份预算，系统应提示用户设置。

### 输入字段

| 字段          | 类型     | 必填 | 说明                |
| ----------- | ------ | -- | ----------------- |
| month       | string | 是  | 预算所属月份，格式 YYYY-MM |
| totalBudget | number | 是  | 月总预算金额            |

### 验收标准

| 编号     | 验收内容                        |
| ------ | --------------------------- |
| AC-001 | 用户输入合法预算后，系统能够保存成功          |
| AC-002 | 用户输入 0、负数或空值时，系统应提示错误       |
| AC-003 | 修改预算后，首页剩余预算、日均预算和城市状态应同步更新 |
| AC-004 | 新月份首次打开应用时，系统应提示用户设置本月预算    |

---

## FR-002 分类预算设置

### 需求描述

系统应支持用户为不同消费类别设置分类预算。分类预算用于展示分类消费进度和触发分类预警。

### 默认类别

系统默认提供以下消费类别：

| 类别编号 | 类别名称 | 说明             |
| ---- | ---- | -------------- |
| C001 | 餐饮   | 吃饭、饮料、水果、外卖等   |
| C002 | 娱乐   | 游戏、电影、演出、会员充值等 |
| C003 | 学习   | 书籍、课程、资料、考试报名等 |
| C004 | 日用   | 生活用品、洗护用品等     |
| C005 | 交通   | 公交、地铁、打车、共享单车等 |
| C006 | 服饰   | 衣服、鞋帽、配饰等      |
| C007 | 医疗   | 药品、挂号、检查等      |
| C008 | 其他   | 无法归入以上类别的支出    |

### 功能规则

1. 分类预算为选填项。
2. 用户可以只设置部分分类预算。
3. 分类预算金额必须大于或等于 0。
4. 分类预算总和允许小于、等于或大于月总预算。
5. 若某类别未设置预算，则该类别不参与分类预算超额预警，但仍统计消费金额。
6. 分类预算修改后，应立即刷新分类进度条和预警判断逻辑。

### 验收标准

| 编号     | 验收内容                 |
| ------ | -------------------- |
| AC-005 | 用户可成功设置某一类别预算        |
| AC-006 | 未设置分类预算的类别仍可正常记账     |
| AC-007 | 修改分类预算后，对应类别进度条应立即更新 |
| AC-008 | 分类预算为空时，系统不应报错       |

---

## FR-003 预算数据保存

### 需求描述

系统应将预算配置保存到本地 Preferences 中。

### 数据保存规则

预算配置包括：

1. 月份。
2. 月总预算。
3. 各分类预算。
4. 最后修改时间。

### 验收标准

| 编号     | 验收内容              |
| ------ | ----------------- |
| AC-009 | 应用关闭后重新打开，预算配置不丢失 |
| AC-010 | 不同月份预算应相互独立       |
| AC-011 | 清除应用数据后，预算配置应被清空  |

# 6.2 记账模块

## FR-004 新增账单

### 需求描述

系统应支持用户新增收入或支出记录。

### 输入字段

| 字段         | 类型       | 必填 | 说明    |
| ---------- | -------- | -- | ----- |
| amount     | number   | 是  | 金额    |
| type       | enum     | 是  | 收入或支出 |
| categoryId | string   | 是  | 消费类别  |
| date       | date     | 是  | 账单日期  |
| remark     | string   | 否  | 备注    |
| createTime | datetime | 是  | 创建时间  |
| updateTime | datetime | 是  | 更新时间  |

### 功能规则

1. 金额必须大于 0。
2. 金额最多支持两位小数。
3. 支出必须选择消费类别。
4. 收入可以选择类别，也可以归为“收入”默认类别。
5. 日期默认取当天日期。
6. 用户可以补记过去日期的账单。
7. 用户不允许选择明显不合理的未来日期，例如超过当前日期 1 年。
8. 新增账单后，系统应立即刷新首页、账单列表、分类看板和小镇状态。

### 验收标准

| 编号     | 验收内容                |
| ------ | ------------------- |
| AC-012 | 输入合法账单后可保存成功        |
| AC-013 | 金额为空、为 0 或负数时不可保存   |
| AC-014 | 支出未选择类别时不可保存        |
| AC-015 | 保存后账单应出现在当月账单列表中    |
| AC-016 | 保存后首页消费数据与小镇状态应自动更新 |

---

## FR-005 编辑账单

### 需求描述

系统应支持用户修改已保存的账单。

### 可修改字段

1. 金额。
2. 类型。
3. 类别。
4. 日期。
5. 备注。

### 功能规则

1. 用户可通过账单详情页或长按账单进入编辑。
2. 修改账单后，应更新 updateTime。
3. 修改涉及金额、类型、类别或日期时，系统应重新计算相关月份的健康度。
4. 如果将账单日期修改到其他月份，原月份和新月份的统计都应重新计算。

### 验收标准

| 编号     | 验收内容                |
| ------ | ------------------- |
| AC-017 | 用户可成功修改账单金额         |
| AC-018 | 修改账单后，列表展示内容同步更新    |
| AC-019 | 修改支出金额后，健康度计算结果同步更新 |
| AC-020 | 修改账单日期跨月时，不同月份统计正确  |

---

## FR-006 删除账单

### 需求描述

系统应支持用户删除账单。

### 功能规则

1. 用户可通过长按账单或账单详情页删除账单。
2. 删除前应弹出二次确认。
3. 删除后账单不再出现在列表中。
4. 删除后应重新计算当前月份统计与小镇状态。
5. 删除操作可采用物理删除，也可采用软删除。

### 推荐实现

建议采用软删除方式：

| 字段         | 说明   |
| ---------- | ---- |
| isDeleted  | 是否删除 |
| deleteTime | 删除时间 |

这样可以为后续版本的“回收站”功能预留空间。

### 验收标准

| 编号     | 验收内容              |
| ------ | ----------------- |
| AC-021 | 用户点击删除后应出现确认提示    |
| AC-022 | 确认删除后账单从列表消失      |
| AC-023 | 删除后首页统计和小镇状态应重新计算 |

# 6.3 账单明细模块

## FR-007 当月账单列表

### 需求描述

系统应按照时间顺序展示当前月份的账单记录。

### 展示字段

| 字段 | 说明          |
| -- | ----------- |
| 日期 | 账单发生日期      |
| 类别 | 消费类别        |
| 金额 | 收入或支出金额     |
| 类型 | 收入 / 支出     |
| 备注 | 用户填写的说明，可为空 |

### 功能规则

1. 默认展示当前月份账单。
2. 支持按日期倒序展示，最新记录在前。
3. 支持按月份切换查看历史账单。
4. 支出金额建议使用负向视觉表达，例如 “-35.00”。
5. 收入金额建议使用正向视觉表达，例如 “+1000.00”。
6. 当账单为空时，应展示空状态提示。

### 验收标准

| 编号     | 验收内容          |
| ------ | ------------- |
| AC-024 | 首页或账单页可显示当月账单 |

# 6.5 财务健康度算法模块

## FR-010 四层动态健康度计算

### 需求描述

系统应根据月总预算、当日消费、本月累计消费和当前日期，计算当前小镇的财务健康度等级。

---

## 6.5.1 基础变量定义

设：

| 变量      | 含义              |
| ------- | --------------- |
| N       | 当前月份总天数         |
| B       | 当前月份总预算         |
| n       | 当前日期为本月第几天      |
| D       | 日均预算            |
| T       | 当前日期对应的累计合理消费指标 |
| E_day   | 当日实际支出          |
| E_total | 本月累计实际支出        |

计算公式如下：

```text
D = B / N
T = n × D
```

其中：

```text
E_day = 当前日期所有支出账单金额之和
E_total = 当前月份所有支出账单金额之和
```

注意：

1. 收入不抵扣支出。
2. 健康度算法只统计支出。
3. 收入可在账单中展示，但不参与城市健康度计算。
4. 若后续版本需要支持“收入抵扣预算”，应作为独立规则新增，不能混入当前算法。

---

## 6.5.2 判定优先级

健康度判定应严格按照以下优先级执行：

```text
第四层：破产状态
第三层：萧条状态
第二层：平稳状态
第一层：繁荣状态
```

即先判断是否破产，再判断是否萧条，再判断是否平稳，最后判断是否繁荣。

---

## 6.5.3 判定规则

### 第一层：繁荣状态

条件：

```text
E_day <= D 且 E_total <= T
```

含义：

用户当天消费没有超过日均预算，并且本月累计消费也没有超过当前日期对应的合理消费线。

表现：

小镇发展良好，建筑繁荣，天气晴朗。

---

### 第二层：平稳状态

条件：

```text
E_day > D 且 E_total <= T
```

含义：

用户当天消费超过了日均预算，但由于之前几天消费较少，本月累计消费仍然没有超过合理进度线。

表现：

小镇保持正常运转，但繁荣程度下降。

---

### 第三层：萧条状态

条件：

```text
E_total > T 且 E_total <= B
```

含义：

用户本月累计消费已经超过了当前日期对应的合理进度线，但尚未超过整月总预算。

表现：

小镇进入风险状态，建筑破旧，天气阴沉。

该规则修复了“某一天不消费就恢复繁荣”的漏洞。只要累计消费没有回到合理均线以内，小镇仍应处于萧条状态。

---

### 第四层：破产状态

条件：

```text
E_total > B
```

含义：

用户本月累计支出已经超过月总预算。

表现：

小镇进入破产状态，建筑变为废墟，天气为雷暴雨。

---

## 6.5.4 伪代码

```typescript
function calculateHealthLevel(
  totalBudget: number,
  monthDays: number,
  currentDayIndex: number,
  todayExpense: number,
  monthExpense: number
): number {
  const dailyBudget = totalBudget / monthDays
  const reasonableTotal = currentDayIndex * dailyBudget

  if (monthExpense > totalBudget) {
    return 4
  }

  if (monthExpense > reasonableTotal && monthExpense <= totalBudget) {
    return 3
  }

  if (todayExpense > dailyBudget && monthExpense <= reasonableTotal) {
    return 2
  }

  return 1
}
```

---

## 6.5.5 边界条件

| 场景             | 处理规则              |
| -------------- | ----------------- |
| 月总预算未设置        | 不计算健康度，提示用户设置预算   |
| 月总预算为 0        | 不允许保存             |
| 当日无支出          | E_day = 0         |
| 当月无支出          | E_total = 0，小镇为繁荣 |
| 当前为每月第一天       | T = D             |
| 当前为每月最后一天      | T = B             |
| E_total 正好等于 B | 不破产，仍根据其他规则判断     |
| E_total 正好等于 T | 不萧条               |
| E_day 正好等于 D   | 不进入平稳，仍可为繁荣       |
| 闰年 2 月         | N = 29            |
| 非闰年 2 月        | N = 28            |

---

## 6.5.6 算法验收标准

| 编号     | 输入条件                       | 期望结果    |
| ------ | -------------------------- | ------- |
| AC-036 | E_total > B                | 健康度 = 4 |
| AC-037 | E_total > T 且 E_total <= B | 健康度 = 3 |
| AC-038 | E_day > D 且 E_total <= T   | 健康度 = 2 |
| AC-039 | E_day <= D 且 E_total <= T  | 健康度 = 1 |
| AC-040 | E_total = B                | 不判定为破产  |
| AC-041 | E_total = T                | 不判定为萧条  |
| AC-042 | E_day = D                  | 不判定为平稳  |

## FR-013 分类预算超额预警特效

### 需求描述

当用户新增一笔支出，且该笔金额直接超过对应分类预算时，系统应触发短暂的局部雷暴特效。

### 触发条件

```text
单笔支出金额 > 对应类别预算
```

示例：

娱乐预算为 500 元，用户新增一笔娱乐支出 600 元，则触发预警。

### 功能规则

1. 仅当类别设置了分类预算时才判断。
2. 未设置分类预算的类别不触发该规则。
3. 收入类型账单不触发该规则。
4. 预警特效持续时间建议为 1.5 至 3 秒。
5. 预警特效不改变当前长期健康度等级。
6. 预警结束后恢复原本小镇状态。

### 验收标准

| 编号     | 验收内容             |
| ------ | ---------------- |
| AC-043 | 单笔支出超过分类预算时触发预警  |
| AC-044 | 单笔支出等于分类预算时不触发预警 |
| AC-045 | 未设置分类预算时不触发预警    |
| AC-046 | 预警结束后恢复正常城市状态    |

4. 首页应让用户一眼看出当前财务状态。
5. 小镇状态变化应直观，不依赖用户阅读复杂文字。
6. 预算设置流程应简单，避免过多配置项。
7. 空状态页面应提供引导按钮，例如“去设置预算”“去记一笔”。

---

### 10.3 可靠性需求

1. 应用异常关闭后，已保存账单不得丢失。
2. 数据库读写失败时，应给出提示。
3. 重复点击保存按钮时，不应产生重复账单。
4. 预算数据读取失败时，应进入安全默认状态。
5. 健康度计算异常时，不应导致应用崩溃。

---

### 10.4 安全与隐私需求

1. 应用不请求网络权限。
2. 所有账单和预算数据仅保存在本地设备。
3. 不上传用户消费数据。
4. 不接入第三方统计 SDK。
5. 不读取通讯录、定位、相册等无关权限。
6. 用户清除应用数据后，本地数据应全部删除。

---

### 10.5 兼容性需求

1. 应支持目标 HarmonyOS 版本。
2. 应适配常见手机屏幕尺寸。
3. 页面布局应支持不同分辨率下正常显示。
4. 图片资源应控制大小，避免低端设备加载卡顿。

---

### 10.6 可维护性需求

1. 健康度算法应独立封装为 `HealthCalculator`。
2. 数据库操作应封装在 Repository 层。
3. 页面不应直接操作数据库。
4. 城市状态映射规则应集中维护，避免散落在多个页面。
5. 类别配置应可扩展，方便后续增加新类别。
6. 关键函数应添加必要注释。

---

## 11. 异常处理需求

| 异常场景        | 处理方式               |
| ----------- | ------------------ |
| 未设置预算进入首页   | 显示引导卡片，提示设置预算      |
| 输入金额为空      | 提示“请输入金额”          |
| 输入金额为 0 或负数 | 提示“金额必须大于 0”       |
| 数据库写入失败     | 提示“保存失败，请重试”       |
| 分类不存在       | 使用“其他”类别兜底，或提示重新选择 |
| 图片资源加载失败    | 显示默认城市图层           |
| 预算读取失败      | 提示重新设置预算           |
| 重复提交账单      | 禁用保存按钮直到保存完成       |
| 当前月份无账单     | 显示空状态页面            |
| 当前月份无预算     | 不展示健康度，提示设置预算      |

## 15. 验收标准总览

系统完成后，应满足以下核心验收条件：

1. 用户可以设置月总预算。
2. 用户可以设置分类预算。
3. 用户可以新增、编辑、删除账单。
4. 用户可以查看当月账单列表。
5. 用户可以查看分类预算消耗进度。
6. 系统可以正确计算日均预算。
7. 系统可以正确计算当前合理消费线。
8. 系统可以正确计算今日支出和本月累计支出。
9. 系统可以根据四层算法正确判断财务健康度。
10. 小镇 UI 可以根据健康度切换四种状态。
11. 单笔消费超过分类预算时可以触发预警特效。
12. 应用关闭后重新打开，数据不丢失。
13. 应用不依赖网络服务。
14. 常规操作不出现明显卡顿。
15. 异常输入有明确提示，不导致应用崩溃。
